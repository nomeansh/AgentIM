# 15 — P1 增强协作与社区功能设计

> **本文档属于 P1 增强 IM 范围。P1 必须建立在 P0 核心 IM 稳定可用之后，不改变 P0 的消息主链路。**

## 概述

P1 的目标不是把 AgentIM 做成另一个 Slack、Telegram 或 WhatsApp，而是在 P0 的云消息底座上补齐三类能力：

1. **组织协作**：工作区、角色、频道分组、任务、知识沉淀。
2. **大规模社区**：超级群组、社区容器、话题分区、公开发现、治理与反垃圾。
3. **人机协作**：Agent 作为受控成员进入频道和私聊，具备审计、审批、记忆和本地执行边界。

P1 完成后，用户故事应从 P0 的“能聊天”升级为：

> 我可以把团队、社群或项目放进 AgentIM，用工作区管理成员与权限，用社区和话题组织大量讨论，用状态、语音房和知识页补充同步/异步协作，并让受控 Agent 在频道内协助完成工作。

## P1 范围边界

### P1 做什么

```
组织协作：
  ✅ Workspace / Team 工作区
  ✅ 工作区成员、角色、邀请、访客
  ✅ 频道分类、默认频道、工作区公告频道
  ✅ 轻量任务、消息转任务、任务提醒
  ✅ Canvas / 知识页：频道内沉淀说明、决策、会议纪要

大规模社区：
  ✅ Community 社区容器：多个群组/频道的上层组织
  ✅ 超级群组：万人级群、慢速模式、成员标签、管理员分工
  ✅ 话题 Topics：大群内分区讨论
  ✅ 消息线程 Threads：围绕单条消息展开支线讨论
  ✅ 入群审批、邀请链接、举报、管理员删除

高级消息与互动：
  ✅ 定时发送
  ✅ 提醒 / 稍后处理
  ✅ 收藏 / 书签 / 集合
  ✅ 自定义表情、贴纸包
  ✅ 快捷回复、自动欢迎语、离线回复
  ✅ 消息自动翻译、语音转文字、摘要

状态与实时协作：
  ✅ Stories / Status 24 小时动态
  ✅ 群/频道语音房（Huddle / Space）
  ✅ 语音房纪要 Artifact（可选 Agent 生成）

隐私与安全：
  ✅ 秘密聊天（端到端加密，服务端不可检索）
  ✅ 阅后即焚 / 定时清理
  ✅ 用户名隐私、手机号隐藏、设备可信校验

Agent 协作：
  ✅ Agent 作为工作区成员
  ✅ Agent 私聊 / 频道提及 / 任务触发
  ✅ Agent 本地执行 daemon 注册
  ✅ Agent 记忆、审批、审计、工具权限
```

### P1 不做什么

| 不做 | 原因 |
|------|------|
| 不做完整应用市场 | 应用市场需要 OAuth、计费、审核、沙箱，放到 P2 |
| 不做 Mini Apps | WebView 容器、支付和第三方权限属于开放平台能力，放到 P2 |
| 不做公域推荐算法 | P1 优先私域协作，公开分发与增长放到 P2 |
| 不做完整视频会议产品 | P1 做轻量语音/视频房入口，底层 WebRTC/SFU 可以外接 |
| 不让 Agent 绕过审批 | Agent 是成员，不是超级管理员 |

## 1. Workspace 工作区

### 设计目标

Workspace 是企业和团队的可选容器。个人用户仍可直接使用 P0 私聊、群组、频道；团队用户可以把成员、频道、任务、知识页和 Agent 统一放入工作区。

### 核心规则

| 规则 | 说明 |
|------|------|
| 一个用户可加入多个 Workspace | 类似 Slack 多工作区 |
| 一个聊天可绑定到一个 Workspace | P1 不做跨 Workspace 共享聊天 |
| Workspace 不替代 P0 Chat | Chat 仍然是消息承载实体 |
| Workspace 权限不能绕过 Chat 权限 | 发送消息仍需通过聊天成员校验 |
| 访客只能访问被授权频道 | Guest 默认不可搜索全员和全频道 |

### 角色模型

| 角色 | 权限 |
|------|------|
| `owner` | 工作区所有权、账单/策略、删除工作区 |
| `admin` | 成员管理、频道管理、角色管理 |
| `member` | 普通成员，可加入公开频道 |
| `guest` | 受限成员，只能访问指定频道 |
| `agent` | Agent 身份，只能执行被授权工具 |

## 2. Community 与超级群组

### Community 容器

Community 是“多个群组/频道的上层集合”，用于学校、俱乐部、开源社区、客户社群等场景。它借鉴 WhatsApp Communities 的“一个公告组 + 多个子群”结构，但仍沿用 AgentIM 的 Chat 模型。

```
Community
  ├─ announcement_chat_id    ← 公告频道，只允许 owner/admin 发言
  ├─ group: general          ← 普通讨论
  ├─ group: help             ← 支持互助
  └─ channel: releases       ← 发布通知
```

### 超级群组

超级群组是 P0 `group` 的增强模式，适用于大规模聊天。

| 能力 | 说明 |
|------|------|
| 万人级成员 | 写路径仍按 chat_id + seq 分片和缓存 |
| 慢速模式 | 成员两次发言间隔限制 |
| 成员标签 | 管理员可给成员打标签，如 `core`、`vip`、`speaker` |
| 加入审批 | 公开申请、邀请链接、管理员审核 |
| 管理员删除 | 管理员可删除违规消息并写审计 |
| 举报队列 | 用户举报消息、成员、社区 |
| 规则确认 | 入群前必须确认社区规则 |

## 3. Topics 与 Threads

### Topics：大群内分区

Topic 是大群内部的“主题聊天室”。它解决 Telegram 大群里消息过载的问题：同一个群可以按 `公告`、`问题求助`、`闲聊`、`版本发布` 分流。

设计约束：
- Topic 只属于 `group` 或 `supergroup`，不属于私聊。
- `im_message.chat_id` 不变，新增 `topic_id` 可空字段。
- chat 级 seq 仍然全局递增，避免破坏 P0 消息同步。
- Topic 有独立未读数和通知设置。

### Threads：围绕消息的支线讨论

Thread 是围绕某条消息展开的回复链，借鉴 Slack 线程。它与 `reply_to_message_id` 不冲突：
- `reply_to_message_id` 表示引用回复。
- `thread_id` 表示该消息属于哪个讨论线程。

```http
POST   /im/chats/{chatId}/topics
GET    /im/chats/{chatId}/topics
PUT    /im/topics/{topicId}
DELETE /im/topics/{topicId}

POST   /im/messages/{messageId}/threads
GET    /im/threads/{threadId}/messages
POST   /im/threads/{threadId}/messages
POST   /im/threads/{threadId}/read
```

## 4. 任务与知识页

### 轻量任务

P1 任务不是完整项目管理系统，只覆盖 IM 内自然产生的待办：
- 从消息创建任务。
- 指派成员或 Agent。
- 设置截止时间、优先级、状态。
- 任务变更回写到来源消息线程。

任务状态：

```
open → in_progress → done
       └──────────→ cancelled
```

### Canvas / 知识页

Canvas 是频道内的轻量知识页，借鉴 Slack Canvas：
- 频道说明、值班表、会议纪要、决策记录、FAQ。
- 支持版本历史。
- 可引用消息、任务、文件、Agent Artifact。
- 不替代独立文档系统，先做 Markdown/富文本 JSON 存储。

```http
POST   /im/messages/{messageId}/tasks
GET    /im/workspaces/{workspaceId}/tasks
PUT    /im/tasks/{taskId}

POST   /im/chats/{chatId}/canvases
GET    /im/chats/{chatId}/canvases
GET    /im/canvases/{canvasId}
PUT    /im/canvases/{canvasId}
GET    /im/canvases/{canvasId}/versions
```

## 5. 状态、Stories 与短生命周期内容

Stories / Status 用于低负担分享，不进入默认聊天时间线。参考 WhatsApp Status、Telegram Stories 和 Signal Stories。

### 规则

| 规则 | 说明 |
|------|------|
| 默认 24 小时过期 | 可配置为 1h / 24h / 7d |
| 可见范围可控 | 联系人、指定群、工作区、公开频道粉丝 |
| 回复进入私聊或频道线程 | 不污染状态列表 |
| 过期后不进搜索 | 只保留必要审计元数据 |
| 高敏感内容禁止公开推荐 | P1 不做公域推荐 |

```http

GET    /im/stories/feed
GET    /im/stories/{storyId}
POST   /im/stories/{storyId}/views
POST   /im/stories/{storyId}/reactions
DELETE /im/stories/{storyId}
```

## 6. 语音房 / Huddle / Space

P1 做“轻量实时房间”，不做完整会议套件。

### 形态

| 房间类型 | 场景 | 角色 |
|----------|------|------|
| `huddle` | 工作区频道临时讨论 | host, speaker, listener |
| `space` | 社区公开语音活动 | host, cohost, speaker, listener |
| `stage` | 发布会 / AMA | host, moderator, speaker, audience |

### 技术边界

- 业务服务只存房间元数据、成员状态、权限和审计。
- 音视频链路交给 WebRTC/SFU 服务，不在业务事务内建立媒体连接。
- 录音、转写、纪要均为异步任务，通过事件触发。
- 纪要可写入 `im_canvas` 或 Agent Artifact。

```http
POST   /im/chats/{chatId}/live-rooms
GET    /im/live-rooms/{roomId}
POST   /im/live-rooms/{roomId}/join
POST   /im/live-rooms/{roomId}/leave
POST   /im/live-rooms/{roomId}/speakers/{userId}
POST   /im/live-rooms/{roomId}/end
POST   /im/live-rooms/{roomId}/summarize
```

## 8. 隐私与安全增强

### 秘密聊天

秘密聊天用于端到端加密场景：
- 只支持私聊和小群，默认不支持频道、超级群、搜索、转发。
- 服务端只保存密文、元数据和必要投递状态。
- 消息内容不可被服务端全文检索。
- 多设备同步需要设备密钥管理，P1 先支持“已授权设备”。

### 阅后即焚

阅后即焚是消息生命周期策略，不等于安全删除保证。

```
发送消息
  → 服务端保存 expire_time
  → 客户端展示倒计时
  → 到期后异步软删除 + 推送 message.expired
  → 搜索索引删除
```

### 用户名与手机号隐私

- 用户可用 `@handle` 被搜索，不暴露手机号。
- 可设置“谁能通过手机号找到我”。
- 可设置“谁能看到在线状态 / 最后在线时间”。
- 新联系人首次私聊走 message request，接受后才成为连接。

| 私聊 Agent | 用户可与 Agent 单独对话 |
| 任务指派 | 任务可分配给 Agent |
| Canvas 写入 | Agent 可在审批后写入知识页 |
| 本地 daemon | 参考 Slock，Agent 可绑定用户自己的计算机执行本地工具 |

### 本地 daemon 约束

```
AgentIM Server
  └→ agent_daemon 注册设备能力
      ├→ repo.read
      ├→ local.shell.readonly
      ├→ local.file.read
      └→ local.file.write（默认审批）
```

要求：
- daemon 必须由用户显式绑定。
- daemon 能力必须声明，不能由 Agent 临时自称。
- 所有本地工具调用写 `agent_tool_call`。
- 任何写文件、执行命令、访问外部凭证的动作必须进入审批或白名单。

## 12. 验收标准

P1 完成的判断标准：

1. 一个团队可以创建 Workspace，邀请成员，建立频道分类和默认频道。
2. 一个社区可以创建公告频道和多个子群，管理员可以处理入群审批和举报。
3. 一个万人级大群可以启用 Topic，成员能按 Topic 阅读、发言和接收未读提醒。
4. 用户可以把消息转成任务、收藏消息、设置提醒、定时发送消息。
5. 用户可以发布 24 小时状态，并控制可见范围。
6. 频道内可以发起语音房，结束后生成记录或 Agent 纪要。
7. 秘密聊天不进入服务端搜索，阅后即焚消息到期后从消息列表和搜索索引删除。
8. Agent 可以作为成员被提及、执行只读任务、请求审批执行高风险动作，并留下完整审计记录。
