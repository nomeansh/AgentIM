# 16 — P2 开放生态与增长功能设计

> **本文档属于 P2 开放生态与增长范围。P2 依赖 P0 核心 IM 和 P1 工作区/社区/Agent 能力，不应反向污染 P0 消息主链路。**

## 概述

P2 的目标是让 AgentIM 从“可私有部署 IM + 协作平台”升级为“可扩展的沟通操作系统”：

1. **开放平台**：Bot API、Mini Apps、OAuth、Webhooks、应用市场。
2. **自动化**：Workflow Builder，把消息、任务、外部事件和 Agent Run 串起来。
3. **增长与公域**：频道发现、创作者工具、公开内容、活动、数据分析。
4. **DevOps 闭环**：Webhook、仓库、PR、CI、AI Review、事故复盘。
5. **企业治理**：SSO、SCIM、DLP、留存、导出、审计、合规。

P2 完成后，用户故事应升级为：

> 我可以把外部系统、机器人、小程序、自动化流程、公开频道和企业治理接入 AgentIM，让沟通、运营、开发、知识和 Agent 执行在同一个平台内闭环。

## 产品参照与取舍

| 参考产品 | 借鉴点 | AgentIM 取舍 |
|----------|--------|--------------|
| Telegram Bot API / Mini Apps | Bot、内联按钮、Web App、支付、订阅、频道生态 | P2 做 Bot API、Mini App 容器、应用授权；支付与订阅可选 |

## P2 范围边界

### P2 做什么

```
开放平台：
  ✅ Bot API v2
  ✅ App Manifest
  ✅ OAuth2 / Personal Access Token / Bot Token
  ✅ Event Subscription
  ✅ Incoming / Outgoing Webhook
  ✅ Mini App WebView 容器
  ✅ 应用目录 / 私有应用安装

自动化：
  ✅ Workflow Builder
  ✅ 消息触发、定时触发、Webhook 触发、Agent 输出触发
  ✅ 条件、分支、审批、重试、失败通知
  ✅ 工作流运行记录与审计

增长与公域：
  ✅ 公开频道目录
  ✅ 频道订阅、频道统计、频道管理后台
  ✅ 公开文章 / 长帖
  ✅ 公开活动 / Space
  ✅ 用户列表 / 内容集合
  ✅ 创作者认证、频道订阅付费（可选）

DevOps：
  ✅ GitHub / GitLab / Jenkins / GitHub Actions / GitLab CI Webhook
  ✅ Repo / Issue / PR / CI 镜像
  ✅ PR 卡片、CI 卡片、失败分析、AI Review
  ✅ 事故频道、复盘 Artifact、发布通知

企业治理：
  ✅ OIDC / SAML SSO
  ✅ SCIM 用户同步
  ✅ DLP 策略、敏感词、外发控制
  ✅ 保留策略、Legal Hold、eDiscovery 导出
  ✅ 全量审计、管理员报表、数据驻留
```

### P2 不做什么

| 不做 | 原因 |
|------|------|
| 不默认启用广告系统 | AgentIM 定位私有部署，广告只作为可选扩展 |
| 不托管第三方代码执行 | Mini App 是前端容器，服务端能力必须走受控 API |
| 不把公开内容流作为核心 | AgentIM 仍以 IM 和协作为中心，公域增长是增强 |
| 不绕过企业审计 | 应用、Bot、Webhook、Agent 都必须进入审计 |
| 不替代 GitHub/GitLab/Jenkins | DevOps 做镜像、通知、自动化与 Agent 协作，不做代码托管 |

## 1. 开放平台

### App Registry

P2 引入应用注册中心，统一管理 Bot、Mini App、Webhook App 和 Workflow App。

| 表名 | 业务含义 | 关键字段 |
|------|----------|----------|
| `open_app` | 应用定义 | app_key, name, app_type, owner_id, status |
| `open_app_manifest` | 应用清单 | app_id, version, scopes, events, commands |
| `open_app_installation` | 应用安装 | app_id, workspace_id, installed_by, status |
| `open_app_token` | 应用令牌 | app_id, token_hash, scopes, expire_time |
| `open_app_oauth_client` | OAuth 客户端 | app_id, client_id, redirect_uris |
| `open_app_oauth_grant` | OAuth 授权 | app_id, user_id, workspace_id, scopes |
| `open_app_event_subscription` | 事件订阅 | app_id, event_type, callback_url, secret |

### App 类型

| 类型 | 说明 |
|------|------|
| `bot` | 可发消息、接收事件、响应命令 |
| `mini_app` | 在 AgentIM 内打开的 WebView 应用 |
| `webhook_app` | 外部系统事件接入 |
| `workflow_app` | 提供 Workflow Action / Trigger |
| `agent_tool` | 为 Agent 提供受控工具能力 |

### Scope 权限

```
message:read
message:write
chat:read
chat:write
workspace:read
workspace:member:read
file:read
file:write
command:register
webhook:receive
workflow:run
agent:tool:invoke
```

原则：
- Scope 最小授权。
- 私有应用只能安装到所属 Workspace。
- 公共应用需要管理员审核后才能进入目录。
- Bot Token、Webhook Secret、OAuth Secret 加密存储。

## 2. Bot API v2

P2 Bot API 面向外部开发者，区别于 P1 Agent：

| 维度 | Bot | Agent |
|------|-----|-------|
| 主要目标 | 程序化消息与集成 | 推理、规划、工具调用 |
| 执行逻辑 | 外部服务控制 | AgentIM Runtime 控制 |
| 权限 | App Scope + 安装范围 | Agent Policy + Approval |
| 记忆 | 由外部服务管理 | AgentIM 可管理结构化记忆 |
| 高风险动作 | 只允许 API 范围内动作 | 需要审批与工具风险检查 |

### Bot 能力

```http
POST /bot/sendMessage
POST /bot/editMessage
POST /bot/deleteMessage
POST /bot/sendPhoto
POST /bot/sendFile
POST /bot/sendPoll
POST /bot/answerCommand
POST /bot/setWebhook
GET  /bot/getUpdates
```

### 交互组件

| 组件 | 用途 |
|------|------|
| Button | 卡片按钮 |
| Select | 选择用户、频道、选项 |
| Modal | 表单输入 |
| DatePicker | 日期选择 |
| FilePicker | 文件选择 |
| MiniAppButton | 打开 Mini App |

所有组件回调必须带 `interaction_id` 和 `idempotent_key`，避免重复提交。

## 3. Mini Apps

Mini App 是在 AgentIM 客户端中运行的嵌入式 Web 应用，参考 Telegram Mini Apps，但按私有部署优先设计。

### 设计规则

| 规则 | 说明 |
|------|------|
| 必须由已安装 App 打开 | 防止任意网页伪装成内部应用 |
| 通过一次性 initData 鉴权 | 服务端签名，短期有效 |
| 只通过 JS Bridge 调用受控能力 | 例如选人、选文件、发消息 |
| 不直接读取本地 Token | 所有凭证走服务端换取 |
| 适配移动端优先 | WebView 首屏必须轻量 |

### JS Bridge 草案

```ts
AgentIM.init()
AgentIM.getContext()
AgentIM.pickUser()
AgentIM.pickChat()
AgentIM.pickFile()
AgentIM.sendMessage()
AgentIM.openInvoice()
AgentIM.close()
```

### API 草案

```http
POST /open/mini-apps/{appId}/sessions
GET  /open/mini-apps/{appId}/config
POST /open/mini-apps/{appId}/callback
```

## 4. Workflow Builder

Workflow Builder 把“触发器 → 条件 → 动作 → 审批 → 结果通知”产品化，参考 Slack Workflow Builder。

### Trigger

| 触发器 | 示例 |
|--------|------|
| `message.created` | 某频道出现关键词 |
| `reaction.added` | 给消息加 ✅ 时创建任务 |
| `schedule` | 每天 09:00 生成日报 |
| `webhook.received` | GitHub PR opened |
| `form.submitted` | Mini App 表单提交 |
| `agent.run.completed` | Agent 产出 Artifact 后通知 |

### Action

| 动作 | 示例 |
|------|------|
| `message.send` | 发送频道通知 |
| `task.create` | 创建任务 |
| `canvas.append` | 追加到知识页 |
| `http.request` | 调外部 API |
| `approval.request` | 发起审批 |
| `agent.run` | 触发 Agent 分析 |

### 表结构

| 表名 | 业务含义 |
|------|----------|
| `workflow_definition` | 工作流定义 |
| `workflow_trigger` | 触发器 |
| `workflow_step` | 步骤 |
| `workflow_run` | 运行记录 |
| `workflow_step_run` | 步骤运行记录 |
| `workflow_approval` | 工作流审批 |

### 执行要求

- 每个 Workflow Run 必须可重放和可审计。
- 外部 HTTP 动作默认需要安装应用授权。
- 失败重试必须有最大次数和退避策略。
- Workflow 调用 Agent 时必须沿用 P1 Agent 权限检查。

## 5. Webhook 与事件总线

Webhook 详见 `08_Webhook与斜杠命令.md`，P2 总设计要求补齐以下平台化能力：

| 能力 | 说明 |
|------|------|
| 统一事件 envelope | 所有事件都有 event_id、event_type、occurred_time、workspace_id |
| 至少一次投递 | 外部 App 必须按 event_id 幂等 |
| 投递日志 | 保存状态码、耗时、重试次数、错误摘要 |
| 手动重放 | 管理员可重放失败事件 |
| 签名校验 | HMAC-SHA256，按 app secret 签名 |

事件示例：

```json
{
  "eventId": "evt_180000000000001",
  "eventType": "message.created",
  "workspaceId": 1001,
  "occurredTime": "2026-05-17T10:00:00Z",
  "data": {
    "chatId": 2001,
    "messageId": 3001
  }
}
```

## 7. 公开内容：Posts、Articles、Lists

P2 可以在频道之上增加公开内容层，但不应该把 AgentIM 变成以信息流为核心的社交网络。

### 内容类型

| 类型 | 说明 |
|------|------|
| `post` | 短动态，可带图片、视频、投票 |
| `article` | 长文，适合公告、教程、复盘 |
| `list` | 用户或频道维护的内容集合 |
| `bookmark` | 用户私有收藏，P1 能力对公开内容扩展 |
| `event` | 公开活动，可关联 live room |

### 规则

- 公开内容必须归属用户、频道或组织。
- 默认按订阅关系分发，不做黑盒推荐。
- 可选“探索页”必须有明确排序规则：最新、热门、编辑精选、标签。
- 公开内容的举报和限流必须先于推荐上线。
- 删除公开内容需要同步删除搜索索引和统计缓存。

## 8. DevOps 闭环

P2 DevOps 详细设计见：
- `08_Webhook与斜杠命令.md`
- `09_代码开发协作流程.md`

本文补充 P2 总目标下的验收闭环：

```
GitHub PR opened
  → Webhook 入站
  → 创建 devops_pull_request 镜像
  → 发送 PR 卡片到 #dev
  → Workflow 判断是否需要 AI Review
  → Agent Run 分析 diff
  → 生成 review artifact
  → 评论 PR 或回写频道
  → CI Webhook 更新状态
  → 失败时触发 Agent 分析日志
```

必须保证：
- 外部系统不可用不影响 P0/P1 IM。
- Webhook 幂等。
- Agent 不能绕过 P1 审批策略。
- PR merge、生产发布默认 `denied` 或 `approval_required`。

## 13. 验收标准

P2 完成的判断标准：

1. 管理员可以注册私有 App，声明 Scope，安装到 Workspace，并接收事件。
2. Bot 可以通过 API 发消息、响应组件回调、注册斜杠命令，且所有动作可审计。
3. Mini App 可以在客户端打开，拿到受签名保护的上下文，并通过 JS Bridge 调用受控能力。
4. 用户可以创建 Workflow，把 Webhook、消息、任务、Agent Run 和通知串联起来。
5. 公开频道可以进入目录，被用户订阅、搜索、统计，并被举报/治理。
6. 公开文章、列表、活动可以发布，但不会破坏 IM 主体验。
7. GitHub/GitLab/CI 事件可以进入 DevOps 镜像表，并在频道内形成卡片通知和 Agent 分析闭环。
8. 企业管理员可以接入 SSO/SCIM，配置留存、DLP、审计导出和 Legal Hold。
9. Agent 工具连接器必须受 App Scope、Agent Policy 和审批策略共同限制。
