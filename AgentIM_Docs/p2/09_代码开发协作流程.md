# 09 — 代码开发协作流程

> **本章属于 P2 DevOps 集成范围，不进入 P0 基础 IM 和 P1 AgentIM 核心。**

## 概述

P2 的目标是让 **IM 沟通 → 任务跟踪 → 代码开发 → CI 验证** 形成完整闭环。想象这样一个场景：

> 你在频道里发现了一个线上故障 → 创建任务 → Agent 分析日志定位根因 → 申请审批修复代码 → 创建 PR → CI 自动执行 → AI Review + 人工 Review → 合并 → 生成复盘报告 → 所有人在频道里看到结果

这就是 P2 要实现的工作流。

## DevOps 核心模型

### Repo（仓库镜像）

`devops_repo` 表存储已连接的外部仓库信息。AgentIM 不托管代码，只做元数据镜像。

| 字段 | 说明 |
|------|------|
| `provider` | `github` / `gitlab` |
| `external_id` | 外部仓库 ID |
| `owner` | 仓库所有者 |
| `name` | 仓库名 |
| `full_name` | 完整名称（如 `org/repo`） |
| `clone_url` | 克隆地址 |
| `default_branch` | 默认分支 |
| `webhook_secret` | Webhook 签名密钥（加密存储） |
| `access_token` | 访问令牌（加密存储） |
| `workspace_id` | 关联的 Workspace |

### Issue（Issue 镜像）

`devops_issue` 存储从外部同步的 Issue。

| 字段 | 说明 |
|------|------|
| `repo_id` | 所属仓库 |
| `external_id` | 外部 Issue ID |
| `title` | 标题 |
| `status` | `open` / `closed` / `reopened` |
| `priority` | 优先级 |
| `labels` | 标签 |
| `assignee_id` | 负责人 |
| `reported_by` | 报告人 |

### Pull Request（PR 镜像）

`devops_pull_request` 存储从外部同步的 PR。

| 字段 | 说明 |
|------|------|
| `repo_id` | 所属仓库 |
| `external_id` | 外部 PR ID |
| `title` | PR 标题 |
| `source_branch` | 源分支 |
| `target_branch` | 目标分支 |
| `status` | `open` / `closed` / `merged` |
| `author_id` | 作者 |
| `review_status` | `pending` / `approved` / `changes_requested` |
| `ci_status` | `pending` / `passed` / `failed` |
| `workspace_id` | 关联的 Workspace |

### CI Run（CI 运行镜像）

`devops_ci_run` 存储 CI 运行记录。

| 字段 | 说明 |
|------|------|
| `repo_id` | 所属仓库 |
| `pr_id` | 关联 PR |
| `external_id` | 外部 CI 运行 ID |
| `status` | `queued` / `running` / `passed` / `failed` / `cancelled` |
| `trigger` | `push` / `pull_request` / `manual` |
| `branch` | 分支 |
| `commit_sha` | Commit SHA |
| `duration` | 运行时长 |
| `logs_url` | 日志 URL |

## 核心工作流：生产故障修复

### 完整场景描述

```
时间线：
10:00 — 线上告警：支付接口超时率飙升
10:01 — 运维在 #incident 频道报告："支付接口 P99 延迟从 200ms 升到 5s"
10:02 — TL 在频道中创建 Task："排查支付接口延迟问题"
10:03 — Task 触发 Agent Run（mode=ACT_WITH_APPROVAL）
        Agent 开始排查：
        Step 1: repo.read("payment/Service.java")
        Step 2: repo.read("payment/Controller.java")
        Step 3: 分析代码 → 发现数据库查询缺少索引
10:05 — Agent 请求审批修改代码
        审批通知 → 频道卡片消息
10:06 — TL 审批通过
10:07 — Agent 修改代码 → 创建 PR
10:08 — CI 自动触发执行
10:09 — CI 通过
10:10 — AI Review 生成检查项 → 无严重问题
10:11 — 人工 Review 并合并
10:12 — Agent 生成复盘 Artifact："支付延迟故障分析报告"
10:13 — 频道通知：故障已修复，附复盘报告
```

### 技术实现

#### 1. 触发 Agent

```java
// 在频道中创建任务时，可选触发 Agent
Task task = taskService.createTask(workspaceId, "排查支付接口延迟问题");
task.setAssigneeType("agent");  // 分配给 Agent

// 创建 Agent Run
AgentRun run = new AgentRun();
run.setWorkspaceId(workspaceId);
run.setTaskId(task.getId());
run.setInstruction("排查支付接口延迟问题：分析 payment 模块代码，定位根因");
run.setMode("ACT_WITH_APPROVAL");
```

#### 2. Agent 代码分析

Agent 通过工具调用分析代码：
- `repo.read` → 读取代码文件
- `repo.search` → 搜索关键字
- `repo.diff` → 查看变更

这些工具调用通过 P1 的 ToolCall 机制记录，审批通过 P1 的 Approval 机制处理。

#### 3. PR 创建

在审批通过后，Agent 调用 `pr.create` 工具：

```java
// Agent 内部调用
PR pr = new PR();
pr.setRepoId(repoId);
pr.setTitle("fix: 添加支付查询索引以解决延迟问题");
pr.setSourceBranch("fix/payment-index");
pr.setTargetBranch("main");
pr.setDescription("在 payment_orders 表的 user_id + status 列添加联合索引");

// Agent 通过 Git API 创建分支、提交代码
repoGateway.createBranch(repoId, "main", "fix/payment-index");
repoGateway.commitFile(repoId, "fix/payment-index", 
    "payment/src/main/resources/db/migration/V2025xxxx__add_index.sql",
    indexSqlContent, "添加支付查询索引");

// 创建 PR
PullRequest createdPR = repoGateway.createPR(repoId, "fix/payment-index", "main",
    "fix: 添加支付查询索引以解决延迟问题", "在 payment_orders 表添加联合索引...");
```

#### 4. CI 集成

CI 通过 Webhook 通知 AgentIM：

```
GitHub Actions 完成
  → POST /devops/webhooks/ci
  → 更新 devops_ci_run.status = "passed"
  → 通过 WebSocket 推送 CI 卡片到频道
  → 频道中显示：✅ CI passed for PR #42
```

#### 5. AI Code Review

PR 创建或更新时触发 AI Review：

```java
POST /devops/pull-requests/{prId}/ai-review

// 返回结构化检查项
[
    {
        "severity": "medium",
        "filePath": "payment/Service.java",
        "lineNo": 42,
        "category": "performance",
        "suggestion": "建议使用批量查询代替循环单条查询",
        "confidence": 0.85
    },
    {
        "severity": "low",
        "filePath": "payment/Controller.java",
        "lineNo": 15,
        "category": "style",
        "suggestion": "方法名建议改为 camelCase",
        "confidence": 0.95
    }
]
```

AI Review **不直接阻断合并**，但 high/critical 级别的检查项会在频道中标记提醒。

### 接口设计

#### Repo 接口

```
GET    /devops/repos                          ← 已连接仓库列表
POST   /devops/repos/connect                  ← 连接新仓库
GET    /devops/repos/{repoId}                 ← 仓库详情
GET    /devops/repos/{repoId}/branches        ← 分支列表
GET    /devops/repos/{repoId}/commits         ← Commit 历史
GET    /devops/repos/{repoId}/files           ← 文件树
GET    /devops/repos/{repoId}/files/content   ← 文件内容
GET    /devops/repos/{repoId}/diff            ← 分支差异
POST   /devops/repos/{repoId}/branches        ← 创建分支
```

#### Issue 接口

```
GET    /devops/issues                         ← Issue 列表
POST   /devops/issues                         ← 创建 Issue
GET    /devops/issues/{issueId}               ← Issue 详情
PUT    /devops/issues/{issueId}               ← 更新 Issue
POST   /devops/issues/{issueId}/link-workspace/{workspaceId}
POST   /devops/issues/{issueId}/link-pr/{prId}
```

#### PR 接口

```
GET    /devops/repos/{repoId}/pull-requests   ← PR 列表
POST   /devops/repos/{repoId}/pull-requests   ← 创建 PR
GET    /devops/pull-requests/{prId}           ← PR 详情
GET    /devops/pull-requests/{prId}/files     ← 变更文件列表
GET    /devops/pull-requests/{prId}/diff      ← Diff 内容
GET    /devops/pull-requests/{prId}/comments  ← 评论列表
POST   /devops/pull-requests/{prId}/comments  ← 添加评论
POST   /devops/pull-requests/{prId}/ai-review ← 触发 AI Review
POST   /devops/pull-requests/{prId}/approve   ← 批准
POST   /devops/pull-requests/{prId}/merge     ← 合并
```

#### CI 接口

```
GET    /devops/ci/runs                        ← CI 运行列表
GET    /devops/ci/runs/{runId}                ← 运行详情
GET    /devops/ci/runs/{runId}/logs           ← 运行日志
GET    /devops/ci/runs/{runId}/tests          ← 测试结果
POST   /devops/ci/runs/{runId}/retry          ← 重跑
POST   /devops/ci/runs/{runId}/analyze        ← 触发 AI 分析失败原因
```

### 与 Workspace 的关系

| 资源 | 与 Workspace 的关系 |
|------|-------------------|
| Repo | 一个 Repo 关联到一个 Workspace |
| Issue | 可关联 Workspace（通过 link 接口） |
| PR | 可关联 Workspace，通过 Repo 间接关联 |
| CI | 通过 PR/Repo 关联到 Workspace |
| Agent Run | 可通过 Issue/PR/CI 触发，受 Workspace 权限约束 |
| 频道通知 | PR/CI 状态变化自动回写到关联频道 |

### 注意事项

```
1. AI Review 不阻断合并
   → Review 结果作为参考信息，不是门禁
   → 合并决策权始终在人工

2. Agent 创建的 PR 必须经过审批
   → 即便是 Agent 生成的代码变更，也走正常 PR Review 流程
   → P1 的审批（approval）和 P2 的 Code Review 是两个独立环节

3. Webhook 必须幂等
   → 网络重试可能导致重复回调
   → provider + external_event_id 唯一约束保证只处理一次

4. 外部系统不可用不能阻塞 IM
   → GitHub 宕机时，AgentIM 的 IM 功能必须正常
   → DevOps 功能的退化应该是优雅的（显示"外部系统不可用"）
```
