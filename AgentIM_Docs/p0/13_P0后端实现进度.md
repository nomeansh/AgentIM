# P0 后端实现进度

更新时间：2026-05-16

## 1. 当前结论

P0 核心 IM 后端能力已按现有设计文档完成一版可编译实现，范围仅包含后端服务、接口契约、数据库建表脚本、认证接入和详细 Javadoc，不包含前端实现。

本轮实现以 `AgentIM-Business` 为核心业务服务，复用 RuoYi-Cloud-Plus 的 Controller -> Service -> Mapper 分层风格、统一响应模型、Sa-Token 登录态和 MyBatis-Plus 数据访问能力。Auth 服务的注册和密码登录流程已切换为调用 Business 模块提供的 IM 账号能力，避免继续依赖原后台用户体系作为 P0 IM 用户主体。

## 2. 已完成范围

### 2.1 微服务与模块接入

- 新增 `AgentIM-Business` 启动入口、Maven 依赖、Nacos/Dubbo/MyBatis 配置和业务资源目录。
- 新增 Business 侧 Dubbo 实现，用于向 Auth 服务提供 IM 用户注册、用户加载、密码校验和默认客户端配置能力。

- 保持前端不实现，不新增前端页面、路由或构建配置。

### 2.2 数据库与领域模型

- 新增 PostgreSQL 16 建表脚本 `AgentIM-Business/src/main/resources/db/im_p0_postgres.sql`。
- 建模覆盖 P0 设计文档中的核心表：用户、联系人、聊天、成员、消息、消息回复、消息反应、已读游标、资源、置顶、投票、投票选项、投票记录、仅自己隐藏消息、审计日志等。
- 新表均包含通用审计字段：`create_by`、`create_time`、`update_by`、`update_time`、`del_flag`。
- 核心唯一约束、幂等键约束、聊天消息序号、成员关系、资源绑定、投票选择等约束已在 SQL 中落地。

### 2.3 P0 业务能力

- 用户系统：注册、登录加载、个人资料查询与修改、用户名搜索、在线状态记录。
- 联系人管理：添加联系人、联系人列表、删除联系人。
- 聊天会话：私聊、群组、频道、保存的消息会话创建与查询。
- 成员管理：添加成员、移除成员、修改角色、查询成员列表。
- 消息能力：发送、编辑、删除、引用回复、转发、按游标拉取消息。
- 消息类型：文本、图片、文件、语音、视频、投票等 P0 类型建模与校验。
- Emoji 反应：添加、删除、按消息聚合展示。
- 已读模型：基于聊天内递增 `seq` 的已读游标、未读数和已读成员列表。
- 文件资源：通过资源服务上传文件，数据库只保存资源元数据和访问控制信息。
- 置顶消息：置顶、取消置顶、列表查询。
- 投票：创建投票消息、查询投票详情、投票与修改投票。
- 保存的消息：注册用户时自动创建保存的消息会话。
- 搜索：当前提供 PostgreSQL `ILIKE` 兜底搜索，并遵守成员可见性和消息删除规则。
- 审计日志：关键写操作记录审计日志。

### 2.4 权限与事务边界

- Controller 采用 `im:*` 权限前缀。
- 业务层通过登录用户、聊天成员、角色和消息归属进行数据级权限校验。
- 私聊限制为两个成员。
- 频道普通成员禁止发言。
- 消息发送使用 `idempotentKey` 防止客户端重试生成重复消息。
- 事务内只做数据库操作；外部 WebSocket 事件通过事务提交后回调发布。
- 文件上传先调用资源服务，再写入资源元数据，避免在数据库事务中执行耗时外部调用。

### 2.5 事件与实时推送

- 新增领域事件发布支持，消息发送、编辑、删除、反应、已读、置顶、投票等操作会构造 IM 事件。
- 默认通过现有 `AgentIM-common-websocket` 能力发布到 Redis/WebSocket 通道。
- 已新增 RocketMQ 事件链路接入面：开启配置后，事务提交后的 IM 事件会先发送到 RocketMQ Topic，再由消费者转发到 WebSocket。
- RocketMQ 默认关闭，便于本地开发和小规模私有部署继续使用 WebSocket 直发；真实环境可通过 Nacos 或本地配置打开。

## 6. 简化方案与剩余风险

### 6.1 搜索能力暂不接入 ES

当前阶段明确暂不接入 Elasticsearch，P0 搜索保留为 PostgreSQL `ILIKE` 数据库搜索实现。

这样做的原因是 P0 目标优先验证核心 IM 闭环：注册、登录、建聊、发消息、已读、资源、投票、审计和 WebSocket 推送。ES 会额外引入索引结构、异步同步、重建索引、分词器、权限过滤一致性、部署运维和故障降级成本。在当前阶段，数据库搜索足够支撑小规模私有部署和功能联调。

保留的能力：

- 可按关键词搜索当前用户可见消息。
- 可限定聊天范围搜索。
- 会过滤已删除消息、仅自己隐藏消息和用户不可见聊天。
- 搜索实现集中在 Mapper/Service 层，当前阶段不引入 ES 依赖。

已知限制：

- `ILIKE` 对大消息量不适合，数据增长后会出现慢查询风险。
- 当前不支持中文分词、相关性排序、高亮、复杂条件组合和附件内容检索。
- 当前搜索强依赖数据库负载，不能把检索压力独立扩容。
- 如果后续引入 ES，需要重新设计消息事件到搜索索引的同步、补偿、重建和权限快照策略。

当前应对：

- P0 阶段优先依赖聊天范围、时间范围和分页限制控制查询成本。
- PostgreSQL 可按实际数据量再补充 `pg_trgm` 或全文索引优化，但本轮不新增依赖和数据库扩展。
- 文档中不再把 ES 作为近期后续步骤，只作为未来规模化搜索的候选方案。

### 6.2 RocketMQ 事件链路

代码层面已写入 RocketMQ 支持，但默认关闭。

启用方式：

- `agentim.im.event.rocketmq.enabled=true`
- `agentim.im.event.rocketmq.topic=agentim-im-event`
- `agentim.im.event.rocketmq.tag=im`
- 同时配置 RocketMQ 标准项：`rocketmq.name-server`、`rocketmq.producer.group`

运行逻辑：

- Service 层仍然只在事务内登记事件。
- 事务提交后，事件发布器优先写入 RocketMQ。
- RocketMQ 消费者收到事件后转发到 WebSocket 分布式通道。
- RocketMQ 未启用时直接 WebSocket 发布。
- RocketMQ 启用但发送失败时，默认回退 WebSocket 直发，避免实时通知完全中断。

剩余风险：

- 当前没有实现事件表或 Outbox，RocketMQ 发送失败后只能靠回退保证实时体验，不能证明可靠事件已持久化。
- 消费失败后的重试、死信队列、重复消费幂等仍需在真实 RocketMQ 环境中验证。
- 当前消费者只负责转发 WebSocket，后续如有审计、搜索索引、离线推送等多消费者，需要拆分 Topic/Tag 或消费组策略。

### 6.3 密码编码简化

密码编码已从开发期 SHA-256 封装切换为 BCrypt 哈希，新注册用户会写入 `bcrypt$` 前缀格式。

实现说明：

- Business 模块已依赖 `AgentIM-common-encrypt`，安全算法来源纳入 common 公共能力体系。
- common-encrypt 的 `EncryptUtils` 主要用于 AES/SM/RSA/Base64 等可逆加解密，不适合作为密码存储方案。
- 密码存储使用 Hutool BCrypt 生成不可逆哈希。
- 旧的 `sha256$` 开发期数据仍可校验，避免本地已有账号立即失效。
- 手机号、邮箱等需要可逆展示的资料字段已接入 `AgentIM-common-encrypt` 的 `@EncryptField`，启用 `mybatis-encryptor` 后由公共模块处理入库加密和查询解密。

剩余风险：

- 当前只做兼容校验，没有自动把旧 `sha256$` 哈希升级写回 BCrypt。
- 密码强度规则仍是注册入参长度校验，尚未统一接入复杂度策略。
- 如果后续引入框架级 PasswordEncoder，需要保留 `bcrypt$` 与 `sha256$` 两类历史前缀兼容。
- `@EncryptField` 依赖 `mybatis-encryptor.enable=true` 和稳定秘钥；如果生产环境更换秘钥，历史手机号和邮箱将无法正常解密。

### 6.4 联调风险

已新增真实联调检查接口：`GET /im/integration/check`，权限标识为 `im:ops:check`。

接口当前检查：

- PostgreSQL：读取 `im_user` 表，验证数据源、schema 和 P0 建表脚本是否可用。
- Redis：写入、读取并删除短 TTL 临时键，验证基础读写和序列化链路。
- RocketMQ：返回启用状态、目标 topic/tag，并在启用时检查 `RocketMQTemplate` 是否初始化。

主要风险包括：

- Nacos 配置项、服务名、Dubbo 分组或版本与运行环境不一致。
- 文件服务 Dubbo 契约在真实上传场景下返回字段与当前假设不一致。
- WebSocket 发布通道、登录用户 ID 和客户端订阅协议需要端到端验证。
- PostgreSQL SQL 方言、JSONB 字段、`UPDATE ... RETURNING` 和索引仍需要在目标数据库上执行完整业务流验证。
- 联调接口不发送真实业务消息，不替代注册、登录、发消息、已读、资源上传等端到端业务联调。

### 6.5 测试风险

当前未新增单元测试或集成测试，后续应优先补充核心行为测试。

优先级最高的测试范围：

- 消息发送幂等：同一 `idempotentKey` 重试不能生成重复消息。
- 权限边界：非成员不能读写聊天，频道普通成员不能发言。
- 私聊约束：私聊必须且只能有两个成员。
- 已读游标：未读数、已读成员列表和游标推进必须一致。
- 投票规则：单选、多选、关闭投票、重复投票覆盖规则。
- 资源访问控制：聊天资源、私有资源和未绑定资源的访问权限。
- 删除语义：仅自己删除与所有人删除必须正确影响消息列表和搜索结果。
