# 产品定位与边界

## 一句话定位

> **AgentIM 是一个开源、可私有部署的即时通讯平台。** 它提供像 Telegram 一样流畅的云消息体验——私聊、群组、频道、富媒体消息、多设备同步——同时保持完全开源，让团队可以自由掌控数据。

## 市场参照与差异

| 产品 | 强项 | 弱项 |
|------|------|------|
| **Telegram** | 云消息、多设备、频道、速度、免费 | 非开源服务端、隐私政策疑虑、企业功能弱 |
| **WhatsApp** | 用户基数大、端到端加密、社区、频道、状态 | 不开源服务器、企业开放能力受限 |
| **Signal** | 强隐私、端到端加密 | 功能简单、无频道、多设备体验差 |
| **Slack** | 团队协作、频道、集成生态 | 价格高、非开源、消息有限制 |
| **Mattermost** | 开源、私有部署、团队协作 | IM 体验不如 Telegram 流畅 |
| **Discord** | 社区、语音、免费 | 不适合严肃沟通、数据不私有 |
| **X** | 公开内容、Spaces、Lists、Bookmarks、创作者分发 | 信息噪声高，社区功能稳定性不足 |
| **Slock** | 人类与 AI Agent 在频道/DM 中协作、本地执行节点 | 新兴产品，适合作为 Agent 协作参考 |
| **AgentIM** | 开源 IM + 云消息 + 频道 + 多设备 + Agent 协作 + 可私有部署 | 生态和成熟度需要时间积累 |

## P0 产品范围

### P0 核心 IM — 做什么

这是 AgentIM 的基石。P0 只有一个目标：**让用户可以用 AgentIM 完成日常即时通讯**。

```
用户系统：
  ✅ 用户注册/登录（复用 RuoYi 认证体系）
  ✅ 用户名（@handle，全局唯一）
  ✅ 用户资料（头像、昵称、个人简介）
  ✅ 在线/离线状态
  ✅ 保存的消息（与自己的私聊）

联系人：
  ✅ 搜索用户（按用户名/昵称）
  ✅ 添加/删除联系人
  ✅ 联系人列表

聊天会话：
  ✅ 私聊（一对一）
  ✅ 群组（多人聊天）
  ✅ 频道（一对多广播）

消息层：
  ✅ 发送文本消息
  ✅ 发送图片/文件（上传 → 引用）
  ✅ 发送语音消息
  ✅ 消息编辑（有时间限制）
  ✅ 消息删除（对所有人 / 仅自己）
  ✅ 引用回复
  ✅ 消息转发（带来源标记）
  ✅ Emoji 反应
  ✅ 投票消息
  ✅ 已读游标 + 未读数
  ✅ 分页查询（游标分页）

实时层：
  ✅ WebSocket 长连接
  ✅ 事件推送：新消息、编辑、删除、反应、已读、在线状态
  ✅ 多设备在线与会话管理

资源层：
  ✅ 文件上传（OSS/MinIO/S3）
  ✅ 图片缩略图
  ✅ 下载鉴权

搜索层：
  ✅ 全局消息搜索
  ✅ 按聊天搜索消息
  ✅ 用户搜索

审计层：
  ✅ 审计日志（消息删除、聊天创建、成员变更）
```

## 重要：不做什么

| 不做 | 原因 |
|------|------|
| 不做端到端加密（P0） | E2EE 与多设备同步、搜索有根本冲突，需要在架构层面单独设计。P0 用 TLS + 服务端加密 |
| 不做视频/语音通话 | 这是专业产品（Zoom、WebRTC）的领域，P0 先做好异步消息 |
| 不做 OA/审批流/CRM | AgentIM 聚焦 IM，不是企业办公套件 |
| 不做完整 CI/CD | CI/CD 由 Jenkins/GitHub Actions 负责 |
| 不做代码托管 | 代码由 GitHub/GitLab 托管 |
| 不做多租户 SaaS | AgentIM 定位私有部署 |

## 关键设计决策

### 为什么消息类型这么丰富？

Telegram 的体验证明：**消息的"内容类型"越多，用户越愿意在平台内完成工作**。P0 支持：
- 文本：最快的方式
- 图片/文件：工作沟通必备
- 语音：比打字快
- 投票：快速达成团队共识

### 为什么不用 workspace 做强制隔离？

- 个人用户不需要 workspace（它只是和朋友聊天）
- 团队用户可以用频道/群组实现隔离
- workspace 作为可选的企业容器延后到 P1

### 为什么消息删除分"对所有人"和"仅自己"？

- **对所有人删除**：发错了、包含敏感信息，发送者有权撤回（有时间限制）
- **仅自己删除**：不想看到某条消息但不影响其他人，属于个人视图范畴
- 两种语义不同，不能用一种覆盖

### 关于搜索

搜索是 IM 的核心功能。P0 直接接入 Elasticsearch 提供全文检索——消息发送后通过 RocketMQ 异步同步到 ES 索引，搜索请求直接查询 ES。
