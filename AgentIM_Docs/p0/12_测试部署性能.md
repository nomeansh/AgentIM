### 集成测试场景（端到端）

```

  1. 非聊天成员尝试发消息 → 拒绝
  2. 非发送者尝试编辑他人消息 → 拒绝
  3. 频道订阅者尝试发言 → 拒绝
```

### 轻量启动模式（开发环境）

降低本地开发的环境门槛——ES 和 RocketMQ 可替换为轻量替代方案：

```text
PostgreSQL 16    → 业务数据库（含全文检索，GIN 索引）
Redis            → 缓存 + Pub/Sub 广播（替代 RocketMQ）
MinIO            → 对象存储
Nacos            → 配置中心
```

```yaml
# application-lightweight.yml — 轻量模式配置
agentim:
  mode: lightweight          # 轻量模式标记
  event:
    broker: redis            # redis 替代 rocketmq
  search:
    backend: postgres        # pg 全文索引替代 es
```

> 生产环境仍然使用 RocketMQ + ES 的完整部署。

### 当前本地 WSL 中间件地址约定

当前联调环境中的中间件都部署在 WSL 内，端口使用各组件默认端口。Windows 侧或宿主侧服务连接中间件时，可优先尝试 WSL IP `192.168.101.6`。

| 中间件 | 访问地址建议 | 默认端口 | 说明 |
|--------|--------------|----------|------|
| PostgreSQL 16 | `localhost:5432` | `5432` | PGSQL 当前比较特殊，联调时使用 `localhost`，不要优先走 WSL IP |
| Redis | `192.168.101.6:6379` | `6379` | 缓存、分布式 session、轻量事件广播 |
| Nacos | `192.168.101.6:8848` | `8848` | 配置中心和注册中心 |
| RocketMQ NameServer | `192.168.101.6:9876` | `9876` | 事件广播启用时配置 `rocketmq.name-server` |
| RocketMQ Broker | `192.168.101.6:10911` | `10911` | Broker 默认通信端口，通常由客户端通过 NameServer 发现 |
| Elasticsearch | `192.168.101.6:9200` | `9200` | HTTP 查询端口；当前 P0 可先保留 PostgreSQL 搜索兜底 |
| MinIO API | `192.168.101.6:9000` | `9000` | 对象存储 API |
| MinIO Console | `192.168.101.6:9001` | `9001` | 对象存储控制台 |

如果 WSL IP 变化，先重新确认 WSL 实际地址，再替换上表中的 `192.168.101.6`；PostgreSQL 仍按当前约定使用 `localhost:5432`。

```

                    │   Nginx/LB   │
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
       ┌──────────┐ ┌──────────┐ ┌──────────┐
       │ Gateway  │ │ Gateway  │ │ Gateway  │
       └────┬─────┘ └────┬─────┘ └────┬─────┘
            │             │            │
       ┌────┴─────┐ ┌────┴─────┐
       │  Auth    │ │ Business │
       └──────────┘ └──────────┘
            │             │
       ┌──────────────────────────────┐
       │  PG16 / Redis / RocketMQ /  │
       │  ES / MinIO                 │
       └──────────────────────────────┘
```

### 部署说明

```
1. P0 可先与业务服务同集群部署
   → AgentIM-Business 可整体水平扩缩容

2. WebSocket 服务可独立部署或与 Business 同进程
   → 分布式部署时，WebSocket 通过 RocketMQ 广播事件（Topic: im-event）
   → ES 独立部署，消息写入后通过 RocketMQ 异步同步索引

3. P0 资源需求：
   → 2 核 4G 起步
   → 推荐 4 核 8G 生产配置
```

## 性能关注点

| 场景 | 风险 | 建议 |
|------|------|------|
| 大聊天消息流 | 分页查询慢 | `chat_id + seq` 复合索引 + 游标分页 |
| 未读数实时计算 | 大群组计算成本高 | Redis 计数器 + 数据库游标兜底 |
| WebSocket 多节点 | session 分布在不同节点 | RocketMQ 广播 Topic + 本地 session 过滤 |
| 大文件上传/下载 | 拖慢接口 | 走对象存储，消息只引用 resourceId |
| 全文搜索 | ES 查询性能 | ES 独立部署，消息通过 RocketMQ 异步索引 |
| 多设备同步 | 同一用户多连接 | 建立 userId→sessionIds 一对多映射 |

### 数据库索引优化（PostgreSQL 16）

```sql
-- 消息列表查询（最频繁的查询）
CREATE INDEX idx_message_chat_seq ON im_message (chat_id, seq);

-- 未读数计算
CREATE INDEX idx_message_chat_status ON im_message (chat_id, status, id);

-- 消息全文搜索（数据库兜底，主力用 ES）
CREATE INDEX idx_message_fts ON im_message
  USING GIN (to_tsvector('simple', content));

-- 幂等检查（部分索引，仅非空时生效）
CREATE UNIQUE INDEX uk_message_idempotent ON im_message (chat_id, idempotent_key)
  WHERE idempotent_key IS NOT NULL;

-- 成员关系
CREATE UNIQUE INDEX uk_chat_user ON im_chat_member (chat_id, user_id);
CREATE INDEX idx_chat_member_user ON im_chat_member (user_id);

-- 已读游标
CREATE UNIQUE INDEX uk_read_state ON im_message_read_state (user_id, chat_id);

-- 联系人
CREATE UNIQUE INDEX uk_contact ON im_user_contact (user_id, contact_user_id);

-- 反应
CREATE UNIQUE INDEX uk_reaction ON im_message_reaction (message_id, user_id, reaction);

-- 置顶
CREATE INDEX idx_pinned_chat ON im_pinned_message (chat_id, pinned_time DESC);
```

### 缓存策略

```text
未读数：
  Redis Key: "im:unread:{chatId}:{userId}" → count
  设置 1 小时 TTL，数据库作为最终来源

聊天列表：
  Redis Key: "im:chats:{userId}" → JSON
  设置 5 分钟 TTL，聊天变更时失效

用户在线状态：
  Redis Key: "im:online:{userId}" → boolean
  WebSocket 连接/断开时更新，设置 2 分钟 TTL（心跳续期）

用户 Session：
  Redis Key: "im:session:{userId}" → sessionId set
  WebSocket 连接/断开时更新
```

## 监控指标

### P0 核心指标

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 消息发送成功率 | 消息写入数据库的成功率 | < 99.9% |
| 消息发送 P95 延迟 | 从请求到写入的延迟 | > 500ms |
| WebSocket 在线连接数 | 当前连接的客户端数 | 用于容量规划 |
| 未读数计算延迟 | 新消息到未读数更新的时间 | > 5s |
| 接口 P95 延迟 | 关键接口的 P95 响应时间 | > 300ms |
| WebSocket 推送延迟 | 消息入库到推送完成 | > 1s |
