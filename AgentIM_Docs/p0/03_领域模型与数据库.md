## 数据模型设计哲学

1. **消息是核心实体，大内容不入消息表** — 文件内容进对象存储，消息正文（`content`）只存文本或摘要。图片/文件/语音/视频都通过 resource_id 引用。
2. **聊天是消息的容器** — 私聊、群组、频道都统一为 `chat` 概念，通过 `type` 区分。不做多表继承。
3. **状态用游标不用状态表** — 已读用游标（每人每聊天一条记录），不做每条消息的已读状态，避免数据爆炸。

## 表详解

### im_user — 用户

用户是系统的基本参与者。继承 RuoYi 的 `sys_user`，扩展 IM 专属字段。

| 字段 | PostgreSQL 类型 | 业务说明 |
|------|----------------|----------|
| `id` | `BIGINT PK` | 雪花 ID（复用 sys_user 的 user_id） |
| `username` | `VARCHAR(50) NOT NULL` | 用户名 @handle（全局唯一，可修改但有限制） |
| `nickname` | `VARCHAR(50)` | 显示昵称 |
| `avatar` | `VARCHAR(500)` | 头像 URL |
| `bio` | `VARCHAR(200)` | 个人简介 |
| `phone` | `VARCHAR(20)` | 手机号 |
| `email` | `VARCHAR(100)` | 邮箱 |
| `status` | `CHAR(1) DEFAULT '0'` | `0`=正常 `1`=停用 |
| `data_scope` | `VARCHAR(30) DEFAULT 'standard'` | 用户级数据访问范围预留：`standard` / `restricted` / `trusted` / `blocked` |
| `permission_tags` | `JSONB DEFAULT '[]'` | 用户权限标签数组预留，用于非聊天业务资源按标签做 allow/deny 匹配 |
| `access_policy` | `JSONB DEFAULT '{}'` | 用户级数据访问策略对象预留，可承载 allowTags、denyTags、resourceScopes、denyReason 等 |

**关键业务逻辑：**
- `username` 是 @提及和搜索的核心标识
- 用户注册时自动创建"保存的消息"聊天（见 im_chat 的 saved 类型）
- 用户删除为逻辑删除，保留聊天记录
- P0 聊天消息、资源、投票等即时通讯数据仍以 `im_chat_member` 做数据权限；`data_scope` / `permission_tags` / `access_policy` 是面向后续非聊天业务资源的用户级策略预留

### im_user_contact — 联系人

用户之间的好友/联系人关系。

| 字段 | PostgreSQL 类型 | 业务说明 |
|------|----------------|----------|
| `user_id` | `BIGINT NOT NULL` | 用户 ID |
| `contact_user_id` | `BIGINT NOT NULL` | 联系人用户 ID |
| `remark` | `VARCHAR(100)` | 备注名 |
| `status` | `CHAR(1) DEFAULT '0'` | `0`=正常 `1`=已删除 |

**唯一约束：** `(user_id, contact_user_id)` 双向表中每条记录只代表一方的关系方向。查询时需双向检查或使用冗余插入（双方各一条）。

**关键业务逻辑：**
- 联系人关系是单向的（A 添加 B 不需要 B 同意），遵循 Telegram 的通讯录模型
- 联系人关系被删除不影响已有聊天记录
- 联系人删除后不再出现在联系人列表中

### im_chat — 聊天会话

聊天的统一容器。私聊、群组、频道、保存的消息都用这张表。

| 字段 | PostgreSQL 类型 | 业务说明 |
|------|----------------|----------|
| `id` | `BIGINT PK` | 雪花 ID |
| `type` | `VARCHAR(20) NOT NULL` | `private` / `group` / `channel` / `saved` |
| `title` | `VARCHAR(100)` | 聊天标题 |
| `avatar` | `VARCHAR(500)` | 群组/频道头像 |
| `description` | `VARCHAR(500)` | 群组/频道简介 |
| `owner_id` | `BIGINT` | 创建者 |
| `seq` | `BIGINT DEFAULT 0` | 聊天内消息序号（从 1 递增） |
| `last_msg_id` | `BIGINT` | 最后一条消息 ID |
| `last_msg_content` | `VARCHAR(200)` | 最后一条消息预览 |
| `last_msg_time` | `TIMESTAMP(6)` | 最后消息时间 |
| `status` | `VARCHAR(20) DEFAULT 'active'` | `active` / `archived` / `deleted` |

**聊天类型说明：**

| 类型 | 业务场景 | 成员上限 | 发言权限 |
|------|----------|----------|----------|
| `private` | 两人私聊 | 2 | 双方可发言 |
| `group` | 多人群聊 | 由配置决定（默认 200） | 所有成员可发言 |
| `channel` | 广播频道 | 无上限 | 仅 owner/admin 可发 |
| `saved` | 保存的消息 | 1（自己） | 仅自己 |

**关键业务逻辑：**
- `private` 类型：双方都是成员，没有显式的 owner
- `group` 类型：owner 可以转让，管理员可以协助管理
- `channel` 类型：普通用户只能订阅（加入），不能发言
- `saved` 类型：用户注册时自动创建，不需要手动操作
- 聊天归档（archived）后不在主列表展示，但消息仍可访问
- `seq` 用于消息排序和游标分页，单调递增

### im_chat_member — 聊天成员

谁在哪个聊天里，以及什么角色。

| 字段 | PostgreSQL 类型 | 业务说明 |
|------|----------------|----------|
| `chat_id` | `BIGINT NOT NULL` | 聊天 ID |
| `user_id` | `BIGINT NOT NULL` | 用户 ID |
| `role` | `VARCHAR(20) NOT NULL` | `owner` / `admin` / `member` / `subscriber` |
| `joined_time` | `TIMESTAMP(6)` | 加入时间 |
| `muted` | `CHAR(1) DEFAULT '0'` | `0`=正常 `1`=免打扰 |

**唯一约束：** `(chat_id, user_id)`

**角色说明：**

| 聊天类型 | owner | admin | member | subscriber |
|----------|-------|-------|--------|------------|
| private | 不适用 | 不适用 | 双方都是 member | — |
| group | 创建者，可转让 | 可管理成员和设置 | 普通成员 | — |
| channel | 频道创建者 | 可发消息和管理 | — | 仅可读 |
| saved | 自己 | — | — | — |

**关键业务逻辑：**
- `private` 聊天的成员不区分角色，都是 `member`
- `channel` 中普通成员的角色是 `subscriber`，不可发言
- group `owner` 离开/删除账号前必须转让 owner

### im_message — 消息

消息是 AgentIM 中最核心、写入最频繁的实体。每条消息都属于一个聊天。

| 字段 | PostgreSQL 类型 | 业务说明 |
|------|----------------|----------|
| `id` | `BIGINT PK` | 雪花 ID |
| `chat_id` | `BIGINT NOT NULL` | 所属聊天 |
| `sender_id` | `BIGINT` | 发送者用户 ID |
| `message_type` | `VARCHAR(30) NOT NULL` | `text` / `image` / `file` / `voice` / `video` / `poll` / `system` / `forward` |
| `content` | `TEXT` | 消息正文 |
| `content_payload` | `JSONB` | 结构化附加数据 |
| `resource_ids` | `JSONB` | 引用的资源 ID 列表 |
| `reply_to_message_id` | `BIGINT` | 引用回复的原消息 ID |
| `forward_from_message_id` | `BIGINT` | 转发来源消息 ID |
| `forward_from_chat_id` | `BIGINT` | 转发来源聊天 ID |
| `forward_sender_id` | `BIGINT` | 转发来源发送者 |
| `client_msg_id` | `VARCHAR(100)` | 客户端消息 ID（去重） |
| `idempotent_key` | `VARCHAR(120)` | 幂等键 |
| `seq` | `BIGINT NOT NULL` | 消息在聊天中的序号 |
| `reply_count` | `INTEGER DEFAULT 0` | 回复数 |
| `status` | `VARCHAR(30) DEFAULT 'normal'` | `normal` / `edited` / `deleted_all` |
| `edit_time` | `TIMESTAMP(6)` | 最后编辑时间 |
| `delete_time` | `TIMESTAMP(6)` | 删除时间 |

**消息类型详细说明：**

| 类型 | content 说明 | content_payload | resource_ids |
|------|-------------|-----------------|-------------|
| `text` | 纯文本 | null | null |
| `image` | 图片描述文本 | 可含宽高、格式 | [resourceId] |
| `file` | 文件描述文本 | 可含文件名、大小 | [resourceId] |
| `voice` | 语音消息文本（自动转写） | 可含时长 | [resourceId] |
| `video` | 视频描述文本 | 可含时长、分辨率 | [resourceId] |
| `poll` | 投票问题 | 投票选项+结果 JSON | null |
| `system` | 系统消息（如"张三加入了群组"） | null | null |
| `forward` | 转发时附带的文本 | 转发元信息 | 可选 |

**关键业务逻辑：**
- 消息发送成功后写入发送者自己的已读游标（发送者默认已读）
- 消息编辑有时间限制（默认 48 小时），只更新 content 和相关字段，不改变 sender
- `deleted_all` = 对所有人删除（发送者可操作，有时间限制），保留消息 ID 占位
- "仅自己删除"通过 `im_user_message_hide` 表实现（见下），不修改消息主表的 status
- 幂等键用于前端防重复提交：相同 chat_id + idempotentKey 只创建一条
- `reply_to_message_id` 指向被回复的原消息

### im_message_reply — 线程/回复

记录回复关系，用于高效查询"某条消息的所有回复"。核心回复关系已在 `im_message.reply_to_message_id` 中冗余一份。

| 字段 | PostgreSQL 类型 | 说明 |
|------|----------------|------|
| `message_id` | `BIGINT NOT NULL` | 回复消息的 ID |
| `reply_to_message_id` | `BIGINT NOT NULL` | 被回复的原消息 ID |
| `chat_id` | `BIGINT NOT NULL` | 聊天 ID |

**唯一约束：** `(message_id)` — 每条回复消息只对应一个原消息

### im_user_message_hide — 用户隐藏消息

记录用户"仅自己删除"的消息。用户隐藏某条消息后，该消息不对该用户显示，但其他人仍然可见。

| 字段 | PostgreSQL 类型 | 说明 |
|------|----------------|------|
| `user_id` | `BIGINT NOT NULL` | 用户 ID |
| `message_id` | `BIGINT NOT NULL` | 隐藏的消息 ID |
| `hidden_time` | `TIMESTAMP(6)` | 操作时间 |

**唯一约束：** `(user_id, message_id)`

**查询示例：**
```sql
-- 查询聊天消息时排除当前用户隐藏的消息
SELECT m.* FROM im_message m
LEFT JOIN im_user_message_hide h
  ON h.message_id = m.id AND h.user_id = :currentUserId
WHERE m.chat_id = :chatId
  AND h.user_id IS NULL           -- 没有被当前用户隐藏
  AND m.status != 'deleted_all'   -- 排除对所有人删除的
ORDER BY m.seq DESC;
```

### im_message_reaction — 消息反应（Emoji）

| 字段 | PostgreSQL 类型 | 说明 |
|------|----------------|------|
| `message_id` | `BIGINT NOT NULL` | 消息 ID |
| `user_id` | `BIGINT NOT NULL` | 用户 ID |
| `reaction` | `VARCHAR(50) NOT NULL` | 反应标识 |

**唯一约束：** 同一条消息、同一个用户、同一个 reaction 只能有一个。

**关键业务逻辑：**
- 同一条消息多个用户可以使用相同 reaction
- 用户可以删除自己的 reaction
- 聊天管理员可以删除任何人的 reaction

### im_message_read_state — 已读游标

使用"游标"模型，每人每聊天一条记录。这个简单的游标是飞书式已读追踪的底层基础。

| 字段 | PostgreSQL 类型 | 说明 |
|------|----------------|------|
| `user_id` | `BIGINT NOT NULL` | 用户 ID |
| `chat_id` | `BIGINT NOT NULL` | 聊天 ID |
| `last_read_message_id` | `BIGINT NOT NULL` | 该用户在该聊天最后阅读到的消息 ID（锚点，用于回查消息） |
| `last_read_seq` | `BIGINT NOT NULL` | 该用户在该聊天最后阅读到的聊天内序号（用于比较和未读计算） |
| `last_read_time` | `TIMESTAMP(6)` | 最后阅读时间 |

> **核心洞察：** 游标不仅用于计算未读数，还天然支持"谁读了哪条消息"——用户读到了 `last_read_seq`，意味着该聊天内序号之前的所有消息都已被阅读。这是飞书式已读追踪的实现基础，无需额外的按消息已读表。

**已读判断逻辑：**

```
用户 U 是否已读消息 M？
  → 查询 U 在该聊天的 last_read_seq
  → 如果 last_read_seq >= M.seq → 已读 ✓
  → 如果 last_read_seq < M.seq → 未读
```

**某条消息的已读状态查询：**

```sql
-- 查询"谁读了消息 M，谁没读"
-- 返回所有聊天成员的已读/未读状态
SELECT cm.user_id,
       rs.last_read_message_id,
       rs.last_read_seq,
       CASE WHEN rs.last_read_seq >= m.seq THEN true ELSE false END AS has_read
FROM im_message m
JOIN im_chat_member cm
  ON cm.chat_id = m.chat_id
LEFT JOIN im_message_read_state rs
  ON rs.user_id = cm.user_id AND rs.chat_id = cm.chat_id
WHERE m.id = :messageId
  AND cm.user_id != :senderId;   -- 排除发送者自己

-- 应用层聚合：
--   readBy:    has_read = true 的成员
--   unreadBy:  has_read = false 的成员
--   readCount:  has_read = true 的计数
--   totalCount: 总成员数（不含发送者）
--   ratio:      readCount / totalCount   ← 用于饼图绿色占比
```

**私聊已读对勾语义：**

| 图标 | 条件 | 含义 |
|------|------|------|
| ✓ (灰) | 消息已写入数据库 | 发送成功 |
| ✓✓ (灰) | 接收方 WebSocket 在线（消息已投递） | 已送达 |
| ✓✓ (绿) | 接收方 `last_read_seq >= message.seq` | 已读 |

**未读数计算方式：**

```sql
SELECT COUNT(1) FROM im_message m
LEFT JOIN im_user_message_hide h
  ON h.message_id = m.id AND h.user_id = :currentUserId
WHERE m.chat_id = :chatId
  AND m.seq > :lastReadSeq
  AND m.status != 'deleted_all'
  AND h.user_id IS NULL;
```

PostgreSQL Upsert 示例（ON CONFLICT）：

```sql
INSERT INTO im_message_read_state (user_id, chat_id, last_read_message_id, last_read_seq, last_read_time)
VALUES (:userId, :chatId, :messageId, :messageSeq, NOW())
ON CONFLICT (user_id, chat_id)
DO UPDATE SET
    last_read_message_id = CASE
        WHEN EXCLUDED.last_read_seq >= im_message_read_state.last_read_seq
        THEN EXCLUDED.last_read_message_id
        ELSE im_message_read_state.last_read_message_id
    END,
    last_read_seq = GREATEST(im_message_read_state.last_read_seq, EXCLUDED.last_read_seq),
    last_read_time = NOW();
```

**唯一约束：** `uk_read_state_user_chat` (user_id, chat_id)

**关键业务逻辑：**
- 游标"只前进不后退"
- 发送消息时自动更新发送者的已读游标（自己发的默认已读）
- 群聊中某成员标记已读后，推送 `read_state.updated` 事件 → 其他成员 UI 实时更新饼图
- 大聊天可以用 Redis 缓存近实时未读数，数据库作为最终一致来源

| `width` | `INTEGER` | 图片/视频宽度 |
| `height` | `INTEGER` | 图片/视频高度 |
| `duration` | `INTEGER` | 语音/视频时长（秒） |
| `access_level` | `VARCHAR(20) DEFAULT 'chat'` | 访问级别 |

**关键业务逻辑：**
- 文件上传独立于消息发送：先上传文件 → 获取 resourceId → 发送消息引用
- 下载时重新鉴权：校验用户是否有对应聊天的访问权限
- `access_level = chat` 表示仅聊天成员可下载（默认）
- 图片上传时自动生成缩略图

## JSON 字段使用原则

| 可以使用 JSON | 不要使用 JSON |
|---------------|---------------|
| 消息结构化载荷（`content_payload`） | 权限主体关系 |
| 资源引用列表 | 成员关系 |
| 工具调用的输入/输出摘要 | 任何需要 WHERE 查询的字段 |
| 动态配置/扩展元数据 | 关联键或授权依据 |
