# 前后端联调修复记录

更新时间：2026-05-16

## 1. 本轮目标

本轮工作目标是对 AgentIM P0 核心 IM 的前端桌面端与后端 Business 模块进行代码 review、联调契约检查、测试验证和问题修复。

执行范围覆盖：

- 后端聊天、成员、消息发送核心服务。
- 前端 WebSocket 事件归并、会话快照同步和消息发送契约。
- 针对联调风险补充回归测试。
- 修复 review 后剩余的幂等键必填与 `chat.updated` 事件 payload 不一致风险。

## 2. 排查方式

本轮使用主线程直接审查与 Codex native subagent 并行排查结合的方式：

- 后端子任务：检查 `ImChatServiceImpl`、`ImMessageServiceImpl` 的聊天退出、成员角色、消息幂等和事务边界。
- 前端子任务：检查 `AgentIM-desktop` 的 WebSocket 事件处理、聊天列表归并和快照回拉。
- Review 子任务：从前后端契约角度确认事件 payload 是否一致、私聊退出是否能让客户端移除本地会话。
- 主线程最终整合：复核所有 findings，补测试，修实现，运行前后端完整验证命令。

## 3. 已修复问题

### 3.1 私聊退出路径错误

问题：

- `deleteChat` 对私聊退出使用普通成员管理路径会触发“该聊天类型不支持成员管理”。
- 私聊的删除语义应是“退出当前用户会话视角”，不应物理删除聊天和历史消息。

修复：

- `ImChatServiceImpl.deleteChat` 对私聊分流到 `leavePrivateChat`。
- `leavePrivateChat` 删除当前用户成员关系，写入审计日志，并向当前用户发送 `chat.deleted`。

涉及文件：

- `AgentIM-Business/src/main/java/com/AgentIM/business/im/service/impl/ImChatServiceImpl.java`
- `AgentIM-Business/src/test/java/com/AgentIM/business/im/service/impl/ImChatServiceImplTest.java`

### 3.2 私聊退出后前端会话残留

问题：

- 早期私聊退出事件使用 `chat.updated` 且 payload 是成员对象。
- 前端会忽略非聊天 payload 并尝试回拉 `getChat`。
- 用户退出后已无读取权限，回拉会失败并被容错吞掉，导致本地聊天列表仍残留该会话。

修复：

- 后端私聊退出改为向退出用户发送 `chat.deleted`。
- 前端处理 `chat.deleted` 时优先使用事件级 `chatId`，避免成员 payload 的 `id` 被误当成聊天 ID。
- 增加前端回归测试覆盖“`chat.deleted` 携带成员载荷时按事件 `chatId` 移除会话”。

涉及文件：

- `AgentIM-Business/src/main/java/com/AgentIM/business/im/service/impl/ImChatServiceImpl.java`
- `AgentIM-desktop/src/stores/imStore.ts`
- `AgentIM-desktop/src/stores/imStore.test.ts`

### 3.3 成员角色边界不严

问题：

- 成员管理接口允许传入 `owner`，存在绕过所有权转让流程的风险。
- 群组和频道角色边界需要按聊天类型收紧：群组只允许 `admin/member`，频道只允许 `admin/subscriber`。

修复：

- `normalizeRole` 禁止成员管理接口设置 `owner`。
- 群组拒绝 `subscriber`，频道拒绝 `member`。
- 空角色仍按聊天类型使用默认值：群组为 `member`，频道为 `subscriber`。

涉及文件：

- `AgentIM-Business/src/main/java/com/AgentIM/business/im/service/impl/ImChatServiceImpl.java`
- `AgentIM-Business/src/test/java/com/AgentIM/business/im/service/impl/ImChatServiceImplTest.java`

### 3.4 消息发送并发幂等冲突

问题：

- 消息发送先按 `idempotentKey` 查询已有消息，再插入新消息。
- 并发重试场景下可能两个请求同时未查到已有消息，其中一个插入成功，另一个触发唯一约束冲突。
- 原逻辑会把冲突直接抛给客户端，未按幂等语义返回已有消息。

修复：

- 捕获 `DuplicateKeyException`。
- 冲突发生后再次按 `chatId + idempotentKey` 回查，若已存在则返回已有消息视图。
- 只有回查仍为空时才继续抛出异常。

涉及文件：

- `AgentIM-Business/src/main/java/com/AgentIM/business/im/service/impl/ImMessageServiceImpl.java`

### 3.5 `idempotentKey` 未强制必填

问题：

- P0 文档要求消息发送必须具备幂等键。
- 早期 `ImMessageSendBo.idempotentKey` 只有长度限制，外部客户端可不传，导致消息发送退化为无幂等保护。

修复：

- 后端入参 `ImMessageSendBo.idempotentKey` 增加 `@NotBlank`。
- `ImMessageServiceImpl.sendMessage` 在服务层再次拒绝空幂等键，保证非 Controller 调用也受保护。
- 前端 `ImMessageSend.idempotentKey` 类型改为必填字段。
- 桌面端 `ComposerBar` 已按 `clientMsgId` 生成 `desktop_${clientMsgId}` 作为幂等键。

涉及文件：

- `AgentIM-Business/src/main/java/com/AgentIM/business/im/domain/bo/ImMessageSendBo.java`
- `AgentIM-Business/src/main/java/com/AgentIM/business/im/service/impl/ImMessageServiceImpl.java`
- `AgentIM-Business/src/test/java/com/AgentIM/business/im/service/impl/ImMessageServiceImplTest.java`
- `AgentIM-desktop/src/types/im.ts`
- `AgentIM-desktop/src/components/ComposerBar.vue`

### 3.6 `chat.updated` payload 契约不一致

问题：

- 后端部分 `chat.updated` 事件发送聊天对象，部分成员操作发送成员对象。
- 前端需要同时兼容两种 payload，容易产生伪聊天、错误回拉和契约歧义。

修复：

- 新增 `buildChatSnapshot`，统一构造 `ImChatVo` 聊天快照。
- `getChat`、聊天创建、归档、取消归档、添加成员、移除成员、修改成员角色等事件统一使用聊天快照。
- 成员移除时，剩余成员收到 `chat.updated` 聊天快照，被移除用户收到 `chat.deleted`。
- `buildChatSnapshot` 使用显式字段映射，避免纯单元测试依赖 `MapstructUtils` 的 Spring 上下文。

涉及文件：

- `AgentIM-Business/src/main/java/com/AgentIM/business/im/service/impl/ImChatServiceImpl.java`
- `AgentIM-Business/src/test/java/com/AgentIM/business/im/service/impl/ImChatServiceImplTest.java`
- `AgentIM-desktop/src/stores/imStore.ts`
- `AgentIM-desktop/src/App.vue`

### 3.7 前端 WebSocket 事件归并问题

问题：

- `chat.updated` 若携带成员对象，前端早期会把成员记录 `id` 当成聊天 ID，写入伪聊天。
- 聊天创建和更新事件到达后，前端缺少服务端快照回拉，成员变更等场景下会话详情可能滞后。

修复：

- 新增 `isChatPayload`，只有具备聊天结构的 payload 才直接合并进聊天列表。
- `App.vue` 增加 `syncChatSnapshot(chatId)`，在 `chat.created` / `chat.updated` 后回拉服务端聊天详情。
- 保留事件处理容错，快照同步失败不打断 WebSocket 事件消费。

涉及文件：

- `AgentIM-desktop/src/stores/imStore.ts`
- `AgentIM-desktop/src/App.vue`
- `AgentIM-desktop/src/stores/imStore.test.ts`

## 4. 新增和调整的测试

### 4.1 后端测试

新增或扩展：

- `normalizeRoleRejectsOwnerFromMemberManagement`
- `normalizeRoleRejectsRolesThatDoNotBelongToTheChatType`
- `normalizeRoleDefaultsByChatType`
- `deletePrivateChatPublishesDeletedEventToCurrentUser`
- `addMemberPublishesChatSnapshotForUpdatedEvent`
- `removeMemberPublishesSnapshotToRemainingMembersAndDeletedEventToRemovedUser`
- `sendMessageRejectsBlankIdempotentKeyBeforeAllocatingSequence`

覆盖点：

- 角色边界。
- 私聊退出事件。
- 成员添加/移除事件契约。
- 空幂等键拒绝。

### 4.2 前端测试

新增或扩展：

- `chat.updated 携带成员载荷时不应误写入伪聊天`
- `chat.deleted 携带成员载荷时应按事件 chatId 移除会话`

覆盖点：

- WebSocket 成员 payload 不污染聊天列表。
- 删除事件使用事件级 `chatId` 清理聊天、消息、置顶和活跃会话。

## 6. 当前结论

本轮 review 与联调修复后，以下风险已关闭：

- 私聊退出后前端会话残留。
- 成员操作产生伪聊天。
- 成员角色越权设置 `owner`。
- 群组/频道角色交叉污染。
- 消息并发重试触发唯一约束后未按幂等返回。
- 外部客户端不传 `idempotentKey` 仍可发送消息。
- `chat.updated` payload 类型不一致。

当前本轮范围内未保留新的已知阻塞风险。

## 7. 后续建议

- 后续如新增移动端或 Web 端，应沿用 `ImMessageSend.idempotentKey` 必填契约。
- 如果未来需要更细粒度成员事件，可新增 `chat.member.added`、`chat.member.removed`、`chat.member.updated`，但不应再让 `chat.updated` 同时承载成员对象和聊天对象。
- 可以在 API 文档中补充“`chat.updated` payload 恒为 `ImChatVo` 快照，`chat.deleted` payload 至少包含 `id/chatId`”的明确契约。
